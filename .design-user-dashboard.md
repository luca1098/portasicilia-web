# User Dashboard - UX/UI Design Document

## Overview

A single-page dashboard for users who book Sicilian tourism experiences. The design mirrors the existing Owner Dashboard patterns (sidebar + header + content area) but with a lighter, consumer-oriented feel. The page is divided into three vertical sections: stats, requests, and bookings.

Route: `/{lang}/dashboard/user`

---

## 1. Page Layout

### Structure

```
+------------------+--------------------------------------------+
| UserSidebar      | DashboardHeader (reuse existing)           |
| (collapsible)    |--------------------------------------------|
|                  | Content area (scrollable)                  |
| - Dashboard      |                                            |
| - My Requests    |   [Stats Cards - 4-col grid]               |
| - My Bookings    |                                            |
|                  |   [Requests Section]                       |
| ----------       |     Status filter pills                    |
| Back to site     |     Requests table / cards                 |
|                  |                                            |
|                  |   [Bookings Section]                       |
|                  |     Bookings table with expandable rows    |
|                  |                                            |
+------------------+--------------------------------------------+
```

### Layout Component

Reuse the existing pattern from `app/[lang]/(dashboard)/dashboard/owner/layout.tsx`:

```
UserLayout
  +-- UserSidebar (new, mirrors OwnerSidebar)
  +-- DashboardHeader (reuse existing)
  +-- <main> scrollable content area
```

### Single-Page Dashboard View

The main dashboard page (`/dashboard/user`) shows everything in one scrollable view:
- Stats cards at the top
- Requests section (only if there are pending/counter-proposed requests)
- Bookings section

The sidebar also links to dedicated `/dashboard/user/requests` and `/dashboard/user/bookings` pages for focused views with full filtering.

### Responsiveness

- **Desktop (lg+)**: Collapsible sidebar (260px / 68px) + full content area. Stats in 4-col grid. Tables with all columns visible.
- **Tablet (md)**: Sidebar hidden by default, toggle via hamburger. Stats in 2-col grid. Tables show key columns, hide secondary ones.
- **Mobile (sm)**: Sidebar as overlay drawer. Stats in 2-col grid. Tables replaced with stacked cards (one card per booking/request).

## 2. Stats Section

Four metric cards in a `grid-cols-2 lg:grid-cols-4` layout, matching the Owner Dashboard card style.

| Card | Label | Value | Icon | Accent |
|------|-------|-------|------|--------|
| Upcoming | "Upcoming" | Count of CONFIRMED bookings with future dates | `CalendarCheck2Icon` | `text-primary` |
| Pending | "Pending Requests" | Count of PENDING_APPROVAL + COUNTER_PROPOSED | `ClipboardListIcon` | `text-chart-2` |
| Completed | "Completed" | Count of COMPLETED bookings | `CheckCircleIcon` | `text-chart-1` |
| Total Spent | "Total Spent" | Sum of all confirmed/completed booking amounts (EUR) | `WalletIcon` | `text-chart-4` |

### Stats API

New endpoint: `GET /bookings/user/stats`

Response:
```json
{
  "upcomingCount": 2,
  "pendingCount": 1,
  "completedCount": 5,
  "totalSpent": "1250.00"
}
```

### Component

`UserDashboardContent` - mirrors `OwnerDashboardContent` structure. Server component fetches stats, passes labels + values as props.

## 3. Requests Section

Shows bookings in request states: `PENDING_APPROVAL`, `COUNTER_PROPOSED`, `REJECTED`.

### Status Filter Pills

Reusable `StatusFilterPills` component (same pattern as Owner). Filters:

| Pill | Filter Value |
|------|-------------|
| All | No filter (show all request statuses) |
| Pending | `PENDING_APPROVAL` |
| Counter-proposed | `COUNTER_PROPOSED` |
| Rejected | `REJECTED` |

### Table Columns

| Column | Content | Width |
|--------|---------|-------|
| Cover | Experience thumbnail (40x40, rounded) | 64px |
| Experience | Experience name (font-medium) | flex |
| Date | Requested date + time slot | auto |
| Participants | Total pax count | auto |
| Status | `StatusBadge` (reuse existing) | auto |
| Actions | Dropdown menu | 64px |

### Counter-Proposal Highlight

When a request has status `COUNTER_PROPOSED`, the row gets special treatment:

1. **Visual indicator**: A subtle left border accent (`border-l-2 border-orange-400`) on the table row.
2. **Inline banner**: Below the main row, an expandable detail area shows:
   - Owner's message (if any), displayed in a quote-style block (`border-l-2 border-muted pl-3 italic`)
   - Proposed alternative dates as selectable cards:
     ```
     +----------------------------------+
     | Proposed alternative dates       |
     |                                  |
     | [Mar 15, 10:00-13:00] [Select]  |
     | [Mar 18, 14:00-17:00] [Select]  |
     | [Mar 22, 10:00-13:00] [Select]  |
     +----------------------------------+
     ```
   - Each proposal card shows: date (formatted), time slot, and a "Select" button
3. **Badge on section header**: When there are counter-proposals, the "Requests" section title shows a notification dot or count badge (e.g., "Requests (1 new)").

### Expanded Row Detail (for all request statuses)

Same grid layout as Owner bookings expanded row:
- Date & time
- Participants breakdown (per type)
- Price breakdown (line items)
- Owner message (if `responseMessage` exists)

### Actions per Status

| Status | Available Actions |
|--------|------------------|
| `PENDING_APPROVAL` | Cancel Request |
| `COUNTER_PROPOSED` | Accept Proposal (with date selection), Cancel Request |
| `REJECTED` | No actions (view only, show owner message) |

### Action: Accept Counter-Proposal

Opens a dialog:
- Title: "Accept Alternative Date"
- Shows the list of proposed dates as radio-button cards
- User selects one
- Confirm button submits the selection
- On success: booking moves to `CONFIRMED`, row moves to Bookings section

### Action: Cancel Request

Opens an `AlertDialog` confirmation:
- Title: "Cancel Request"
- Description: "Are you sure you want to cancel this booking request?"
- Cancel / Confirm buttons
- On success: booking moves to `CANCELLED`, removed from view

## 4. Bookings Section

Shows bookings in confirmed/final states: `CONFIRMED`, `COMPLETED`.

### Table Columns

| Column | Content | Width |
|--------|---------|-------|
| Cover | Experience thumbnail (40x40, rounded) | 64px |
| Experience | Experience name (font-medium) | flex |
| Date | Booking date + time slot | auto |
| Participants | Total pax count | auto |
| Total | Price total (formatted EUR) | auto |
| Expand | Chevron toggle button | 64px |

### Expandable Row Detail

Triggered by clicking the row (same as `OwnerBookingsTable > BookingRow`):

```
+-------------------------------------------------------+
| Date & Time          | Participants    | Payment       |
| Mar 15, 2026         | 2x Adult        | Total: 150EUR |
| 10:00 - 13:00        | 1x Child        | Deposit: 50EUR|
|                      |                 |               |
| Price Breakdown                                       |
| 2x Adult ticket - 100EUR                              |
| 1x Child ticket - 50EUR                               |
+-------------------------------------------------------+
```

Three-column grid on desktop (`grid-cols-1 sm:grid-cols-2 lg:grid-cols-3`), stacking on mobile.

### Actions per Status

| Status | Available Actions |
|--------|------------------|
| `CONFIRMED` (future date) | Cancel Booking (if cancellation policy allows) |
| `CONFIRMED` (past date) | None (will transition to COMPLETED) |
| `COMPLETED` | Leave Review (if no review yet) |

### Action: Cancel Booking

Opens an `AlertDialog`:
- Title: "Cancel Booking"
- Description: explains refund policy based on experience's cancellation policy
- Shows refund amount (if applicable)
- Cancel / Confirm buttons

### Action: Leave Review

Opens a `Dialog` with:
- Star rating (1-5)
- Text area for review body
- Submit button
- (This can be a separate component: `ReviewDialog`)

## 5. Notifications / Messages

### Counter-Proposal Attention System

Counter-proposals are the most important notification for users. They require action and have time sensitivity.

**Level 1 - Section badge**: The "Requests" section heading shows a count badge when there are `COUNTER_PROPOSED` bookings:
```
Requests  [2 new]
```
Style: small orange pill badge inline with the section title.

**Level 2 - Row highlight**: Counter-proposed rows have:
- `border-l-2 border-orange-400` left accent
- The `StatusBadge` already uses `bg-orange-100 text-orange-800`
- Auto-expanded to show the proposal details on first load (when there are 3 or fewer counter-proposals)

**Level 3 - Owner message callout**: When `responseMessage` is present on any request:
```
+----------------------------------------------+
| Message from [Owner Name]:                   |
| "The requested date is fully booked. I can   |
|  offer these alternative dates instead."     |
+----------------------------------------------+
```
Style: `bg-muted/50 rounded-lg p-3 border-l-2 border-orange-300` with the owner's avatar (small, 24px) next to their name.

### Dashboard Header Bell

The existing `DashboardHeader` already has a bell icon with a red notification dot. For the user dashboard:
- Show the dot when there are unread counter-proposals or status changes
- Future enhancement: clicking opens a notification dropdown

## 6. Empty States

Each section has a dedicated empty state, matching the Owner Dashboard pattern (`rounded-xl border border-border bg-card p-8 text-center`).

| Section | Condition | Message |
|---------|-----------|---------|
| Stats | No bookings at all | Stats show all zeros (no special empty state) |
| Requests (all) | No requests | "You have no pending requests. Browse experiences to book your next adventure!" + CTA button linking to `/experiences` |
| Requests (filtered) | No requests for selected filter | "No [status] requests found." |
| Bookings (all) | No confirmed/completed bookings | "No bookings yet. Once your requests are confirmed, they will appear here." |
| Counter-proposals | Counter-proposed request with no proposals data | Fall back to standard status badge display |

### First-Time User (Zero State)

When the user has never made any booking:
- Stats cards show zeros
- Single centered empty state instead of both sections:
  ```
  [Compass/Map illustration or icon]
  "Your Sicilian adventure starts here"
  "Browse experiences and book your first trip."
  [Explore Experiences] (primary button)
  ```

### Component Details

#### `UserSidebar`
- Mirrors `OwnerSidebar` exactly in structure
- Items: Dashboard (home), My Requests, My Bookings
- Footer: "Back to PortaSicilia" link
- Base path: `/{lang}/dashboard/user`

#### `UserDashboardContent`
- Receives: `title`, `welcome`, `stats` (labels), `values` (formatted strings)
- Renders: header, stats grid, then conditionally renders a preview of requests (if any pending/counter-proposed) and recent bookings
- On the main dashboard page, shows max 5 recent items per section with a "View all" link

#### `UserRequestsTable`
- Client component (`'use client'`)
- Props: `bookings`, `initialNextCursor`, `fetchParams`, `activeStatus`
- Contains: `StatusFilterPills`, table with rows, load-more button
- Each row can expand to show details + counter-proposals
- Counter-proposed rows are visually distinct

#### `UserBookingsTable`
- Client component (`'use client'`)
- Props: `initialBookings`, `initialNextCursor`, `fetchParams`
- Contains: table with expandable `BookingRow` components, load-more button
- Expanded rows show date, participants, payment details, price breakdown

#### `AcceptProposalDialog`
- Props: `open`, `onOpenChange`, `booking` (with counterProposals), `onSuccess`
- Renders proposed dates as selectable cards with radio-button behavior
- Submit calls `acceptProposalAction(bookingId, selectedProposalIndex)`

#### `CancelBookingDialog`
- Props: `open`, `onOpenChange`, `bookingId`, `onSuccess`
- Simple AlertDialog confirmation
- Calls `cancelBookingAction(bookingId)`

#### `OwnerMessageBlock`
- Props: `message`, `ownerName`, `ownerAvatar`
- Renders a styled quote block with the owner's message
- Used inside expanded request rows

#### `CounterProposalCard`
- Props: `date`, `timeSlot`, `selected`, `onSelect`
- Small card showing formatted date and time range
- Highlighted border when selected

## 8. Mobile Adaptations

### Tables to Cards

On screens below `md`, tables transform into stacked card lists:

```
+----------------------------------------+
| [Cover Image]  Experience Name         |
|                Mar 15, 2026            |
|                10:00 - 13:00           |
| 2 participants    [Status Badge]       |
|                                        |
| [Counter-proposal banner if relevant]  |
| [Actions button]                       |
+----------------------------------------+
```

Implementation: Use responsive utility classes to hide `<Table>` on mobile and show a card-based layout instead. This can be handled with `hidden md:block` on the table and `block md:hidden` on the card list.

### Counter-Proposal Cards on Mobile

On mobile, counter-proposal details are shown as full-width stacked cards below the request card, rather than inside a table expanded row:

```
+----------------------------------------+
| Owner suggests:                        |
| "I can offer these dates..."           |
|                                        |
| [Mar 15 | 10:00-13:00]  [Accept]     |
| [Mar 18 | 14:00-17:00]  [Accept]     |
+----------------------------------------+
```

## 9. Data Flow

### Server Components (Pages)

Each page is a server component that:
1. Gets the session via `getServerSession(authOptions)`
2. Redirects to `/{lang}` if not authenticated
3. Fetches data via `apiServer` with auth headers
4. Passes data + translation labels to client components

### Client Components (Tables/Dialogs)

Client components handle:
- State management (expanded rows, dialogs, loading states)
- Cursor-based pagination ("Load more" button)
- Optimistic status updates after actions
- Uses `useAction` hook for all async operations

### API Endpoints Needed

| Endpoint | Method | Description |
|----------|--------|-------------|
| `GET /bookings/user` | GET | List user's bookings (paginated, filterable by status) |
| `GET /bookings/user/stats` | GET | User dashboard stats |
| `POST /bookings/{id}/accept-proposal` | POST | Accept a counter-proposal (body: `{ proposalIndex: number }`) |
| `POST /bookings/{id}/cancel` | POST | Cancel a booking/request |
| `POST /bookings/{id}/review` | POST | Submit a review (body: `{ rating: number, body: string }`) |
