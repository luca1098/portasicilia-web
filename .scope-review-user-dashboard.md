# Scope Review: User Dashboard

**Reviewer role:** Devil's Advocate / Quality Reviewer
**Date:** 2026-03-06
**Spec:** "Simple, single-page dashboard showing requests with info, bookings with info, and messages (e.g., date change from owner)."

---

## 1. What Is Clearly IN Scope

- **Single page** at `app/[lang]/(dashboard)/dashboard/user/page.tsx` — no sub-routes, no sidebar, no layout beyond what already exists.
- **List of user bookings** fetched via `GET /bookings/my` (endpoint exists, role-gated to USER).
- **All booking info displayed**: experience name/cover, date, time slot, participants, status, amounts (total/deposit), contact info, price breakdown via `priceSnapshot`.
- **Owner messages visible inline**: when status is `COUNTER_PROPOSED` or `REJECTED`, display the `responseMessage` field. When status is `COUNTER_PROPOSED`, display the `counterProposals` JSON (proposed alternative date/time).
- **User actions on counter-proposals**: "Accept" button calling `PATCH /bookings/:id/accept-counter` (endpoint exists).
- **Cancel booking**: button calling `PATCH /bookings/:id/cancel` (endpoint exists).
- **Status badges**: reuse existing `components/dashboard/bookings/status-badge.tsx`.
- **i18n**: all labels in both `it.json` and `en.json`.

## 2. What Is OUT OF Scope — Do NOT Build These

| Item | Why it is out of scope |
|---|---|
| **Sidebar navigation** | Spec says "single page". The owner dashboard has a sidebar because it has sub-routes (requests, bookings, stats). The user does not need one. |
| **Sub-routes** (e.g., `/user/bookings`, `/user/requests`) | Spec says "single page dashboard". Everything on one page. |
| **Stats cards** (total bookings, spend, etc.) | Not mentioned in spec. The owner has stats because they run a business. The user just wants to see their bookings. Don't copy the owner pattern blindly. |
| **Real-time notifications / WebSocket** | Not mentioned. The user sees messages when they open the dashboard. Polling or SSR refresh is sufficient. |
| **Chat / messaging system** | The spec says "messages" but the context is owner responses (responseMessage, counterProposals). This is NOT a chat feature. Do not build a messaging UI. |
| **Filtering / search / pagination** | A typical user will have a handful of bookings. A simple list sorted by date is enough. Add filtering later only if needed. |
| **Review/rating system** | Not in spec. |
| **Booking detail page** | Not in spec. Show all info inline in the list. If the list gets cluttered, use an expandable accordion or modal — not a separate route. |

## 3. Risks and Concerns

### RISK 1: Backend endpoint does not return `responseMessage` or `counterProposals`

**Severity: HIGH**

The `findUserBookings` method in `booking.service.ts` (line 819) uses:
```
include: {
  experience: { select: { id, name, slug, cover } },
  priceSnapshot: true,
}
```
It does NOT explicitly select `responseMessage`, `counterProposals`, `respondedAt`, or `timeSlot`. Since Prisma returns all scalar fields by default, `responseMessage` and `counterProposals` WILL be included. However, `timeSlot` (a relation) is NOT included, so the user will not see which time slot their booking is for.

**Action required:** The backend `findUserBookings` must add `timeSlot: true` to the include block so the user can see their booking's time slot. This is a one-line backend change.

### RISK 2: No frontend API client for `GET /bookings/my`

**Severity: MEDIUM**

There is no `lib/api/user-bookings.ts` file. The owner dashboard has `lib/api/owner-bookings.ts` and `lib/actions/owner-bookings.actions.ts`. Equivalent files need to be created for the user side. This is straightforward but must be done.

### RISK 3: No frontend type for user booking response

**Severity: MEDIUM**

The existing `BookingResponse` type in `lib/api/bookings.ts` does NOT include `responseMessage`, `counterProposals`, `createdAt`, `status` change fields, or `cover` on the experience. A proper `UserBooking` type must be defined that matches what the backend actually returns.

### RISK 4: "Messages" scope creep

**Severity: HIGH**

The spec says "in case of message (e.g., for a date change from owner) the user should visualize all." This is ambiguous. The existing data model supports:
- `responseMessage`: a single text message from the owner when they respond to a booking
- `counterProposals`: a JSON array of alternative dates when the owner counter-proposes

This is NOT a message thread. It is a one-shot response. The implementation should show this as an inline alert/banner on the booking card when present — not as a separate "messages" section, not as a chat, not as a notification center.

If the product team wants back-and-forth messaging, that is a separate feature requiring new backend models and endpoints. Flag this and push back.

### RISK 5: Over-engineering by copying owner dashboard patterns

**Severity: MEDIUM**

The owner dashboard has: stats cards, two sub-pages (requests vs bookings), a sidebar, server actions, cursor-based pagination. The user dashboard needs NONE of this for MVP. The temptation to copy the owner's architecture must be resisted. A single server component page with a client component list is sufficient.

## 4. Recommended Approach

### Architecture (minimal)

```
app/[lang]/(dashboard)/dashboard/user/page.tsx     -- server component, fetches bookings
components/dashboard/user/user-dashboard-content.tsx -- client component, renders list
components/dashboard/user/user-booking-card.tsx      -- individual booking card
lib/api/user-bookings.ts                            -- API client (GET /bookings/my)
lib/actions/user-bookings.actions.ts                -- server actions (cancel, accept-counter)
```

No sidebar. No layout changes. No sub-routes. No stats.

### UI Structure (single page)

1. **Header**: title + welcome message (already exists)
2. **Booking list**: cards or rows, each showing:
   - Experience name + cover image
   - Date + time slot
   - Participants + total amount
   - Status badge (reuse existing component)
   - If COUNTER_PROPOSED: show owner's message + proposed dates + Accept/Decline buttons
   - If REJECTED: show owner's message
   - If PENDING_APPROVAL / CONFIRMED: show Cancel button
3. **Empty state**: friendly message when no bookings exist

### Backend Change Required

In `be/src/booking/booking.service.ts`, `findUserBookings` method — add `timeSlot: true` to the include block:
```typescript
include: {
  experience: { select: { id: true, name: true, slug: true, cover: true } },
  priceSnapshot: true,
  timeSlot: true,  // <-- ADD THIS
}
```
